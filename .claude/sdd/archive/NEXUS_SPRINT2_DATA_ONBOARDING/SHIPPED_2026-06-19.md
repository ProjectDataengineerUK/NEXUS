## What Was Built

### Multi-tenancy & Onboarding (P0)
- **`manifest.yml`**: `references:` block com 3 tabelas consumer (`customer_table`, `transactions_table`, `events_table`) + `privileges: EXECUTE TASK`
- **`setup_script.sql`**: `RAP_ORG_ISOLATION` aplicada em 11 tabelas, `CONFIG.DATA_SOURCES`, External Access Integration, 3 Tasks com RESUME
- **`0_Setup.py`**: Wizard 3-step (Map Tables → Manage Users → Configure APIs)
- **`utils/onboarding.py`**: 6 funções utilitárias para o wizard

### Canonical Tables & Revenue (P1)
- `CORE.ACCOUNTS`, `CORE.PRODUCTS`, `CORE.INTERACTIONS` com RAP + demo data
- `MART.REVENUE_OPPORTUNITY_SCORE` + `SP_SCORE_REVENUE_OPPORTUNITIES` + `DT_REVENUE_OPPORTUNITY_SCORE`
- Dynamic Tables: `DT_EXECUTIVE_KPIS`, `DT_CUSTOMER_HEALTH`, `DT_REVENUE_MOVEMENT`

### RBAC Granular (P1)
- 6 `APPLICATION ROLE` por agente: Executive, Revenue, Customer, Risk, Data Steward, Operations
- Least-privilege GRANTs: cada agente só acessa as tabelas do seu domínio

### KBS (P1)
- Schema `KBS` com `DOCUMENTS`, `SOURCES`, `SEARCH_LOGS`
- `CORTEX SEARCH SERVICE KB_SEARCH_SERVICE` com filtro `kb_name`
- Loaders: Snowflake Core docs (14 páginas) + Cortex AI docs (12 páginas)
- Chunker: `chunk_document` + `chunk_markdown` com overlap

### Airflow Pipelines (P1 — provider-side)
- `salesforce_ingest_dag.py`: Account/Contact/Opportunity/Lead → STAGING
- `zendesk_ingest_dag.py`: tickets/users/orgs → CORE.TICKETS via MERGE
- `stripe_ingest_dag.py`: customers/subscriptions/invoices/charges → CORE.SUBSCRIPTIONS + CORE.TRANSACTIONS
- `kbs_refresh_dag.py`: weekly refresh, troca documentos >8 dias

### 6º Agent + P2 (P1/P2)
- `operations_agent.yaml`: Cortex Analyst + Cortex Search + SQL exec (read-only)
- `11_Sales_Intelligence.py`: Pipeline de oportunidades + Revenue Movement

- `terraform/modules/monitoring`: Resource Monitors + freshness task

## Lessons Learned

### ✅ O que funcionou bem

**Reúso de objetos existentes**  
`CONFIG.ORG_USER_MAP` já existia com `(org_id, user_name)` — foi extendida com `ALTER TABLE ADD COLUMN IF NOT EXISTS role`. O SP `CORE.REGISTER_REFERENCE` também já existia com `SYSTEM$SET_REFERENCE`. Reusar economizou tempo e evitou inconsistências.

**RAP fallback para empty state**  
Usar `OR NOT EXISTS (SELECT 1 FROM CONFIG.ORG_USER_MAP)` como fallback no RAP foi a decisão certa: o Native App funciona corretamente na instalação inicial antes de qualquer usuário ser mapeado.

**DESIGN document com code patterns copy-paste**  
O DESIGN incluiu 10 padrões prontos para copiar. Isso eliminou ambiguidade durante o build e acelerou a escrita de cada arquivo.

**Dual-track orchestration claro**  
Separar explicitamente "Tasks Snowflake = consumer-side no setup_script" vs "Airflow DAGs = provider-side" evitou confusão sobre onde cada pipeline roda.

### ⚠️ O que melhorar

**Numeração de setup files pode colidir**  
`snowflake/setup/11_*.sql` e `12_*.sql` já existiam de sprints anteriores com outros nomes. Sprint 3 deve definir uma convenção de numeração mais clara (ex: `sprint2_11_*.sql`) ou usar prefixo de versão.

**STAGING schema não está no setup_script**  
Os DAGs Airflow criam tabelas `STAGING.*` com `CREATE TABLE IF NOT EXISTS` no runtime. Melhor adicionar o schema STAGING ao `setup_script.sql` para que ele exista antes da primeira execução do DAG.

**KBS loaders dependem de HTML scraping**  
A qualidade dos chunks depende de regex para limpar HTML — frágil. Sprint 3 deve usar `beautifulsoup4` ou a API REST oficial da Snowflake Docs se disponível.

**Testes SQL são manuais**  
Os testes em `tests/sql/` precisam ser executados manualmente no Snowflake após instalação. Integrar com `snowflake-cli` + script de CI seria o próximo passo natural.

**Operations Agent sem semantic model**  
`operations_agent.yaml` referencia `operations_model.yaml` em um stage que ainda não existe. Sprint 3 precisa criar o semantic model YAML para o agente poder usar Cortex Analyst.
