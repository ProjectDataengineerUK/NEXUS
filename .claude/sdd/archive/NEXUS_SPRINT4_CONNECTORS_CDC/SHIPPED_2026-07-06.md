## What Was Built

### Conectores (P0/P1)
- **`sap_ingest_dag.py`**: extrai `Customers`/`Invoices`/`Orders` via SAP OData (evita RFC/BAPI — sem dependência do SAP NW RFC SDK proprietário)
- **`oracle_ingest_dag.py`**: extrai via `oracledb` em modo thin (puro Python, sem Oracle Instant Client)
- **`hubspot_ingest_dag.py`**: extrai `contacts`/`deals`/`companies` via API v3 com paginação `after` cursor
- Todos os 3 seguem o padrão TaskFlow API dos conectores do Sprint 2 (retry, `SnowflakeHook`, `snowflake_nexus` conn_id), mas só extraem e carregam `STAGING.*` — sem MERGE em Python

### CDC via Streams (P0/P1 — redesenhado)
- 9 tabelas `STAGING.<FONTE>_<OBJETO>` (org_id + raw_data VARIANT + loaded_at)
- 9 Streams `APPEND_ONLY` capturando linhas novas
- 9 Tasks (`SCHEDULE = '15 MINUTE'`, `WHEN SYSTEM$STREAM_HAS_DATA(...)`) fazendo MERGE incremental para `CORE.CUSTOMERS`, `CORE.TRANSACTIONS` ou `CORE.ACCOUNTS` conforme o objeto
- IDs prefixados por fonte (`SAP-`, `ORCL-`, `HS-`) evitando colisão de namespace

### Documentação (P2)
- **`airflow/README.md`**: tabela de todos os 6 conectores + Airflow Variables necessárias por fonte + pré-requisitos (SAP Gateway, oracledb thin mode, escopos do HubSpot Private App token)

## Lessons Learned

### ✅ O que funcionou bem

**Corrigir o DEFINE durante o DESIGN evitou trabalho perdido**
A premissa original de Stream+Task entre CORE e MART teria conflitado com o motor de Dynamic Tables (que já faz refresh incremental nativo). Identificar isso na fase de DESIGN — antes de escrever qualquer SQL — evitou construir algo que quebraria ou seria redundante.

**Reaproveitar o padrão exato dos conectores existentes**
Os 3 novos DAGs replicam a estrutura de `stripe_ingest_dag.py` (extract → `load_to_staging` com `executemany` + `PARSE_JSON`) quase literalmente, só removendo a etapa de merge em Python (que virou responsabilidade da Task). Isso manteve consistência de estilo e reduziu risco de bugs novos.

**org_id como coluna de staging, não extraído do payload**
Primeira versão do design errou ao tentar extrair `org_id` do JSON da fonte (`raw_data:OrgId`) — SAP/Oracle/HubSpot não têm esse conceito nativamente. Corrigido para `org_id` ser uma coluna própria da staging, preenchida pelo Airflow via `Variable.get("NEXUS_DEFAULT_ORG_ID")`, replicando exatamente como Stripe/Zendesk já fazem.

**CI/CD verde de primeira nos dois commits do build**
Diferente da sessão anterior de hardening de CI/CD (que precisou de ~20 rounds de correção), os dois commits deste sprint passaram no `Native App — dev` sem nenhuma iteração — resultado direto de ter uma pipeline confiável (Sprint 3) e um DESIGN bem verificado antes do BUILD.

### ⚠️ O que melhorar

**Testes ainda são estruturais, não de execução real**
`TestSAPDAG`/`TestOracleDAG`/`TestHubSpotDAG` validam que o código existe e segue o padrão esperado (grep em source), não que os DAGs rodam corretamente contra uma fonte real. Mesma lacuna já identificada no Sprint 3 para o Cortex Analyst — um ambiente de staging com fontes mockadas (ex: SAP Gateway sandbox, Oracle XE local, HubSpot sandbox account) fecharia essa lacuna em um sprint futuro.

**Terraform para segredos do Airflow continua em aberto para todos os 6 conectores**
Nenhum conector (nem os do Sprint 2) tem as credenciais gerenciadas via Terraform — todas dependem de configuração manual do Airflow. Se isso virar dor operacional real (múltiplos ambientes, rotação de secrets), vale um sprint dedicado a isso, não um item avulso de P2.
