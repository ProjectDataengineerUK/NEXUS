# 1. O problema que o produto resolve

Grandes empresas têm o mesmo conjunto de dores, independentemente do setor:

1. **Dados espalhados**
   CRM, ERP, atendimento, billing, contratos, documentos, logs, marketing, supply chain, transações, IoT, RH, financeiro.

2. **IA desconectada da realidade operacional**
   Muitos pilotos de IA ficam bonitos em demo, mas não acessam os dados corretos, não respeitam governança e não chegam à produção.

3. **Executivos querem respostas, não SQL**
   CFO, COO, CMO, CRO, Head de Risco e Head de Operações querem perguntar:  
   “Por que a margem caiu?”  
   “Quais clientes vão churnar?”  
   “Quais contratos têm risco?”  
   “Onde estamos perdendo dinheiro?”  
   “O que devo fazer hoje?”

4. **Governança e compliance travam inovação**
   Financeiro, saúde, telecom e indústria não podem simplesmente jogar dados em ferramentas externas.

5. **Cada área compra ferramentas separadas**
   BI, data quality, catalog, RAG, IA para documentos, copiloto, detecção de anomalias, previsão, atendimento, governança. Isso vira custo, retrabalho e baixa adoção.

Endpoints úteis:

```text

https://data.sec.gov/api/xbrl/companyconcept/CIK0000320193/us-gaap/Revenues.json
```

Exemplo de uso:

```python
import requests

headers = {
    "User-Agent": "NexusAI jonataslimac@gmail.com"
}

url = "https://data.sec.gov/api/xbrl/companyfacts/CIK0000320193.json"

```

import requests
import pandas as pd
from snowflake.connector.pandas_tools import write_pandas
import snowflake.connector

def fetch_world_bank(country="BRA", indicator="NY.GDP.MKTP.CD"):
    url = f"https://api.worldbank.org/v2/country/{country}/indicator/{indicator}?format=json"
    response = requests.get(url)
    data = response.json()[1]
    return pd.DataFrame(data)

def load_to_snowflake(df, table_name):
    conn = snowflake.connector.connect(

        account="ACCOUNT",
        warehouse="COMPUTE_WH",
        database="NEXUS_DEMO",
        schema="RAW_MARKET"
    )
    write_pandas(conn, df, table_name)

## 17.3 Score de risco simples

```sql
CREATE OR REPLACE TABLE MART.COMPANY_RISK_SCORE AS
SELECT
    c.company_id,
    c.company_name,
    COALESCE(n.negative_news_count, 0) * 0.4
    + COALESCE(f.revenue_decline_score, 0) * 0.4
    + COALESCE(m.market_volatility_score, 0) * 0.2
    AS risk_score
FROM MART.COMPANY_360 c
LEFT JOIN MART.NEWS_RISK n ON c.company_id = n.company_id
LEFT JOIN MART.FINANCIAL_RISK f ON c.company_id = f.company_id
LEFT JOIN MART.MARKET_RISK m ON c.company_id = m.company_id;
```

### Exemplo

Seu cliente quer perguntar:

```text

Revenue tables

Cortex Analyst

```

**Melhor escolha:** Snowflake.

**Melhor escolha:** Databricks.

# 3. Como cada um cobra

## Snowflake

Você paga principalmente por:

1. **Compute** via créditos.
2. **Storage**.
3. **Cloud services**, conforme uso.
4. Recursos adicionais conforme edição, região e funcionalidades.

Snowflake informa que o preço depende da edição escolhida e do consumo de storage e compute. ([Snowflake](https://www.snowflake.com/en/pricing-options/?utm_source=chatgpt.com))

### O que faz Snowflake ficar caro?

- Warehouses ligados sem necessidade.
- Consultas ruins.
- Muitos dashboards atualizando toda hora.
- Falta de auto-suspend.
- Transformações pesadas mal desenhadas.
- Uso intenso de IA sem controle.
- Duplicação de dados.
- Falta de monitoramento por área.

### Como controlar custo

- Auto-suspend em warehouses.
- Warehouses separados por workload.
- Resource monitors.
- Query optimization.
- Dynamic Tables com cuidado.
- Materialized views quando fizer sentido.
- Caching.
- Limites por usuário/agente.
- Logs de custo por pergunta de IA.

---

## Databricks

Você paga principalmente por:

1. **DBUs** da Databricks.
2. **Máquinas/VMs** do cloud provider.
3. Storage no data lake.
4. Serverless compute, se usar.
5. Workloads específicos: jobs, SQL, ML, notebooks, serving etc.

A Databricks usa modelo pay-as-you-go e oferece descontos com compromissos de uso. ([Databricks](https://www.databricks.com/product/pricing?utm_source=chatgpt.com)) No Azure Databricks, por exemplo, existem workloads diferentes como Jobs Compute, Jobs Light Compute e All-Purpose Compute. ([Microsoft Azure](https://azure.microsoft.com/en-us/pricing/details/databricks/?utm_source=chatgpt.com))

### O que faz Databricks ficar caro?

- Clusters esquecidos ligados.
- All-purpose clusters usados para jobs.
- Jobs mal particionados.
- Spark mal otimizado.
- Muitos notebooks exploratórios.
- Autoscaling mal configurado.
- Serverless sem limites.
- Equipe sem experiência em Spark/FinOps.

### Como controlar custo

- Job clusters em vez de all-purpose.
- Autoscaling bem configurado.
- Photon quando fizer sentido.
- Delta optimization.
- Spot instances, quando aceitável.
- Cluster policies.
- Budgets por workspace.
- Separação dev/stage/prod.
- Monitoramento de DBU por workload.

# 13. Segurança: regra obrigatória para MCP Agents

MCP é poderoso, mas pode ser perigoso se você expõe ferramentas demais.

Há alertas recentes sobre riscos de segurança em MCP, especialmente quando servidores permitem execução de comandos, filesystem, Git ou ferramentas sensíveis sem allowlist e isolamento. Relatos recentes destacaram riscos de RCE, prompt injection e composição insegura de MCP servers. ([Tom's Hardware](https://www.tomshardware.com/tech-industry/artificial-intelligence/anthropics-model-context-protocol-has-critical-security-flaw-exposed?utm_source=chatgpt.com))

Para seu produto, use estas regras:

## Regras mínimas

```text
1. Preferir read-only por padrão.
2. Separar agente analítico de agente executor.
3. Usar allowlist de tools.
4. Nunca expor warehouse admin para agentes comuns.
5. Usar roles específicas por agente.
6. Logar toda chamada MCP.
7. Exigir aprovação humana para ações externas.
8. Bloquear comandos shell arbitrários.
9. Não conectar filesystem sensível em produção.
10. Validar prompts contra prompt injection.
```

## Roles Snowflake por agente

```text
AGENT_EXECUTIVE_READONLY
AGENT_DATA_ENGINEER_DEV
AGENT_GOVERNANCE_AUDITOR
AGENT_REVENUE_ANALYST
AGENT_DOCUMENT_READER
AGENT_ACTION_OPERATOR
```
